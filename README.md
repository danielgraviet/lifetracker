# lifetracker# Vibe Check — Personal Like/Dislike Activity Tracker

A personal life-logging system that uses a **Telegram bot** to collect daily voice memos, an **LLM** to parse them into structured activity entries, and a **web dashboard** to surface patterns over time. The goal: understand what energizes you and what drains you, so you can make better life and career decisions.

---

## How It Works

```
You (voice memo) → Telegram Bot → Whisper API (transcription) → Claude API (parsing) → Database → Dashboard
```

1. Every evening, a Telegram bot sends you a nudge: *"Hey — what did you enjoy or not enjoy today?"*
2. You reply with a voice memo (30 seconds to a few minutes of natural speech).
3. The system transcribes the audio using OpenAI's Whisper API.
4. The transcript is sent to Claude with a structured extraction prompt.
5. Claude returns an array of activity entries with sentiment, tags, energy effect, and context.
6. Entries are stored in a database.
7. A web dashboard lets you browse, search, and visualize patterns over time.

## Data Model

### Entry (primary record)

| Field               | Type                                         | Description                                                      |
| ------------------- | -------------------------------------------- | ---------------------------------------------------------------- |
| `id`                | UUID                                         | Primary key                                                      |
| `date`              | DATE                                         | The day this activity occurred                                   |
| `activity`          | TEXT                                         | Short name of the activity (e.g., "code reviews")                |
| `sentiment`         | ENUM: `liked`, `disliked`, `mixed`           | How you felt about it                                            |
| `intensity`         | INT (1–5)                                    | How strongly you felt (1 = mild, 5 = extreme)                   |
| `energy_effect`     | ENUM: `energizing`, `draining`, `neutral`    | Separate from sentiment — you can dislike something energizing   |
| `category`          | TEXT                                         | Broad life area (see Category list below)                        |
| `tags`              | TEXT[] (array)                                | Freeform tags generated by the LLM, guided by tag vocabulary     |
| `context`           | TEXT                                         | Short note capturing relevant context from your words            |
| `source_transcript` | TEXT                                         | Original transcript segment this entry was parsed from           |
| `memo_id`           | UUID (FK → Memo)                             | Links back to the original voice memo                            |
| `created_at`        | TIMESTAMP                                    | When this record was created                                     |

### Memo (raw voice memo record)

| Field              | Type      | Description                                    |
| ------------------ | --------- | ---------------------------------------------- |
| `id`               | UUID      | Primary key                                    |
| `date`             | DATE      | Date the memo was recorded                     |
| `telegram_file_id` | TEXT      | Telegram's file reference for the voice message |
| `transcript`       | TEXT      | Full Whisper transcription                     |
| `duration_seconds` | INT       | Length of the voice memo                       |
| `created_at`       | TIMESTAMP | When this record was created                   |

### TagVocabulary (evolving tag dictionary)

| Field        | Type      | Description                                     |
| ------------ | --------- | ----------------------------------------------- |
| `tag`        | TEXT (PK) | The canonical tag string                        |
| `usage_count`| INT       | How many entries use this tag                   |
| `first_seen` | DATE      | When this tag first appeared                    |
| `last_seen`  | DATE      | Most recent use                                 |
| `aliases`    | TEXT[]    | Alternate phrasings the LLM should normalize    |

### Starting Categories

These are broad and will evolve. The LLM should use these as defaults but can suggest new ones:

- `work` — job tasks, meetings, projects
- `health` — exercise, sleep, medical
- `social` — time with people, events
- `creative` — art, writing, music, side projects
- `learning` — reading, courses, skill-building
- `chores` — errands, admin, household
- `leisure` — relaxation, entertainment, hobbies

## LLM Parsing Prompt

The core extraction prompt lives in `prompts/parse_entry.txt`. Here is the spec for what it should do:

**Input:** A raw transcript of a voice memo about the user's day.

**Output:** A JSON array of entry objects.

**Prompt behavior requirements:**

1. Extract every distinct activity the user mentions.
2. For each activity, infer sentiment, intensity, energy effect, category, and tags.
3. Use the provided tag vocabulary for consistency, but create new tags when nothing fits.
4. Write a short `context` field that captures the *why* behind the sentiment — not just restating the activity.
5. If the user's language is ambiguous, lean toward `mixed` sentiment rather than guessing.
6. Return valid JSON only — no preamble, no markdown fences.

**Example input:**
```
Spent the morning doing code reviews, which honestly I kind of loved — felt like mentoring.
Then had a 2-hour planning meeting that completely drained me. Oh, and I went on a bike
ride after work which was great.
```

**Example output:**
```json
[
  {
    "activity": "code reviews",
    "sentiment": "liked",
    "intensity": 4,
    "energy_effect": "energizing",
    "category": "work",
    "tags": ["code-review", "mentoring", "engineering"],
    "context": "Enjoyed the mentoring aspect of reviewing others' code"
  },
  {
    "activity": "planning meeting",
    "sentiment": "disliked",
    "intensity": 4,
    "energy_effect": "draining",
    "category": "work",
    "tags": ["meetings", "planning"],
    "context": "Two hours felt excessive and was energy-draining"
  },
  {
    "activity": "bike ride",
    "sentiment": "liked",
    "intensity": 3,
    "energy_effect": "energizing",
    "category": "health",
    "tags": ["cycling", "outdoors", "exercise"],
    "context": "Post-work ride was a positive way to end the day"
  }
]
```

## API Endpoints

### Entries

| Method | Path                      | Description                              |
| ------ | ------------------------- | ---------------------------------------- |
| GET    | `/entries`                | List entries (paginated, filterable)     |
| GET    | `/entries/:id`            | Get a single entry                       |
| POST   | `/entries`                | Manually create an entry (bypass bot)    |
| PUT    | `/entries/:id`            | Edit an entry (fix tags, sentiment, etc.)|
| DELETE | `/entries/:id`            | Delete an entry                          |

**Query parameters for `GET /entries`:**
- `date_from`, `date_to` — date range filter
- `sentiment` — filter by liked/disliked/mixed
- `category` — filter by category
- `tags` — comma-separated tag filter (entries matching ANY tag)
- `energy_effect` — filter by energizing/draining/neutral
- `search` — full-text search across activity, context, and transcript
- `page`, `per_page` — pagination (default 20 per page)

### Analytics

| Method | Path                          | Description                                       |
| ------ | ----------------------------- | ------------------------------------------------- |
| GET    | `/analytics/sentiment-trend`  | Sentiment distribution over time (weekly buckets)  |
| GET    | `/analytics/top-tags`         | Most frequent tags, filterable by sentiment        |
| GET    | `/analytics/energy-map`       | Activities plotted by sentiment × energy           |
| GET    | `/analytics/category-breakdown` | Entries per category over time                  |
| GET    | `/analytics/day-of-week`      | Patterns by weekday                                |

### Tags

| Method | Path                  | Description                              |
| ------ | --------------------- | ---------------------------------------- |
| GET    | `/tags`               | Full tag vocabulary with usage counts    |
| PUT    | `/tags/:tag`          | Add alias or merge tags                  |

### Weekly Summary

| Method | Path                      | Description                                  |
| ------ | ------------------------- | -------------------------------------------- |
| GET    | `/weekly-summary`         | Get current week's summary                   |
| GET    | `/weekly-summary/:date`   | Get summary for week containing given date   |

## Key Design Decisions

**Sentiment and energy are separate dimensions.** You can dislike something that's energizing (a hard gym session you dread) or like something that's draining (hosting a big dinner). Both dimensions matter for understanding what to pursue.

**Tags are LLM-generated, not predefined.** The system maintains a growing vocabulary to keep the LLM consistent, but doesn't restrict it. Your personal taxonomy should emerge organically from your language.

**Source transcripts are always kept.** As prompts improve or models get better, you can re-parse old memos to extract richer data. The raw transcript is the ground truth.

**The bot is single-user.** This is a personal tool. The `TELEGRAM_USER_ID` env var restricts all interactions to your account. No auth system needed beyond that.

**Corrections happen in Telegram.** If the LLM misparsed something, you should be able to reply in the chat to fix it — no need to open a dashboard just to change a tag.

## Getting Started (for the coding agent)

1. Start with Phase 1. Get the Telegram bot running locally with a `/start` command and voice memo handling.
2. Use `python-telegram-bot` v20+ (async).
3. Set up a PostgreSQL database with the schema described above. Use Alembic for migrations from the start.
4. Follow the project structure exactly as specified — it's designed for clean separation of concerns.
5. Write the extraction prompt in `prompts/parse_entry.txt` as a standalone text file, not inline in code. The parser should load it at runtime.
6. When building the dashboard, use Vite + React. Keep it simple — no state management library needed initially, just React hooks and fetch.
